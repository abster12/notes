# Notes

## Teaching preferences

- **Article format:** paragraphs for concepts, bullets only for lists/steps, ASCII diagrams for architecture/flow, tables for comparisons. NOT walls of text.
- **Code samples** where they make a concept concrete (tool schemas, SQL, config). Java for the legacy, Python for the new.
- **One thing per article.** If it has 3 sub-topics, those are 3 articles.
- **Cross-link aggressively.** This workspace is part of a system design collection (sister: learn-sde3-staff). Every article that references a vault concept should link to it.
- **The "why" matters more than the "what."** A reader should leave understanding the principle, not memorizing a list.

## Style

- Conversational, not academic
- Use "I" / "you" / "we" — the reader is a person, not a class
- Lead with the meta-reason, not the technical detail
- Cut filler ("it is important to note that..."). Cut adverbs.
- "Don't" is fine. Plain English is fine.

## Out of scope (user preference)

- No code samples longer than 20 lines — break them up or put them in a code block with explanation
- No external links in the body unless they're a primary source. Reference list goes in RESOURCES.md.
- No emoji unless the user asked.
